# Terraform

This section introduces **Terraform** for infrastructure as code (IaC). You will learn how to define infrastructure declaratively, parameterise it with variables, and manage lifecycle changes in a controlled, auditable way.

The focus is on understanding the Terraform workflow end-to-end: initialise, plan, apply, verify, and destroy.

If unfamiliar with Terraform, the [Terraform registry](https://registry.terraform.io/providers/hashicorp/azurerm/latest/docs) is the ultimate guide on what is possible and how to frame the code blocks and what the dependencies are.

For general Terraform syntax help, HashiCorp has excellent resources [here](https://developer.hashicorp.com/terraform/tutorials/configuration-language/resource). The CLI is also important to understand how to complete the tasks. Details of what to run are included in the tasks, but for further reading, please see [here](https://developer.hashicorp.com/terraform/tutorials/cli).

**Unlike PowerShell, Terraform is case sensitive**

## Important Note on State

The Terraform state file is a JSON blob that records the details and configuration of all infrastructure managed by Terraform. It acts as a single source of truth for what the infrastructure should look like. Any changes to Terraform-managed infrastructure should always be made through code, not the portal or GUI — this prevents configuration drift. Terraform will always attempt to reconcile real infrastructure with its known state, so manual changes made outside of Terraform can cause conflicts or be silently reverted.

In real scenarios, the state file **should** be stored remotely — for example, in an Azure Storage Account. This ensures that all engineers share the same view of the infrastructure. With local state, there is a substantial risk of drift: if Engineer A applies a change locally, Engineer B's state file has no knowledge of it and may revert those changes on their next apply.

In this module, everything that is built will also be destroyed. Resource names should include the engineer's own name (for example, `RG-Geoff-Test-001`) to avoid clashes between participants. No subscription IDs or any secrets should be committed to this repository.

The location of this is configured within the `backend.tf` 

## Why This Matters

Terraform is a core tool for creating repeatable infrastructure. It reduces manual drift, improves collaboration, and makes changes easier to review before they are applied.

## Setup

Install Terraform with Chocolatey (run in a PowerShell terminal):
`choco install terraform -y`
Verify installation:
`terraform version`

An Azure subscription where you can create and destroy resources is also required.

### Extensions

1. **HashiCorp Terraform** (hashicorp.terraform) - core Terraform language support and validation.
2. **Azure Terraform** (ms-azuretools.vscode-azureterraform) - Azure-focused Terraform assistance.
3. **Terraform Graph** (saramorillon.terraform-graph) - visualise resource dependencies.
4. **Terraform** (4ops.terraform) - additional Terraform editing support.

## Learning Goals

- Create and use providers, resources, and variables.
- Separate configuration into sensible files (`main.tf`, `variables.tf`, `outputs.tf`).
- Use plans to inspect changes before apply.
- Understand state basics and why it matters.
- Build and remove a simple environment safely.
