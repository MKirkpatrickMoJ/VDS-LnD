# GitHub Actions

**GitHub Actions** is GitHub’s integrated automation and CI/CD platform. It enables repositories to define workflows that build, test, scan, and deploy code in response to repository events such as pushes, pull requests, or scheduled triggers. Workflows are defined as YAML files located in the `.github/workflows` directory, ensuring full version control and traceability alongside the source code they support.

Each workflow executes on a runner, which may be a GitHub‑hosted virtual machine or a self‑hosted machine managed by a team or organisation. Workflows are composed of jobs, and each job contains a series of steps that run individual actions—reusable automation components that perform tasks such as checking out code, installing dependencies, performing static analysis, authenticating to cloud platforms, or deploying applications and infrastructure. This structure provides a consistent, repeatable approach to CI/CD and general automation across projects.

GitHub Actions supports complete automation pipelines, including unit testing, security scanning, artifact production, packaging, and deployment to various environments. By colocating automation with repository code and configuration, GitHub Actions facilitates transparent, collaborative, and auditable workflow management suitable for software engineering, platform operations, infrastructure‑as‑code, and other automated processes.

## Note

GitHub Actions **have** to be defined within the `./github/workflows` folder. To keep that folder free from clutter, the tasks will be here within markdown. As actions on appear once they enter the default branch of a repository, all required files will exist, but will need modifications. The modifications will only be accessible after pushing the working branch to GitHub, then they can be run from the specific branch.

## Why This Matters

GitHub Actions is a key enabler for team delivery standards: repeatability, traceability, and controlled deployment. Instead of running critical processes manually on local machines, pipelines provide consistent automation with a clear history.

## Extensions

1. **YAML** (redhat.vscode-yaml) - YAML formatting and schema validation.
2. **GitHub Actions** (me-dutour-mathieu.vscode-github-actions) - autocomplete and validation for workflows.
3. **GitHub Actions** (github.vscode-github-actions) - GitHub-integrated workflow tooling from VS Code.
4. **GitLens** (eamodio.gitlens) - useful when reviewing automation changes over time.

## Learning Goals

- Understand workflow structure: triggers, jobs, steps, and actions.
- Separate validation/build tasks from deployment tasks.
- Use reusable actions and shared patterns where appropriate.
- Keep workflow files readable and maintainable.
- Build confidence in making safe CI/CD changes.
