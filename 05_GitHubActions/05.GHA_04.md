## Scenario

Reusable workflows allow you to define a workflow once and call it from multiple places with different inputs. This reduces duplication and makes maintenance easier. The template workflow handles the logic, and the caller workflow dispatches it with the appropriate parameters.

Additionally, the use of GitHub environments allows the same variable name to be used for different environments providing different values.

## Task

1. Open the caller workflow at `./.github/workflows/05-task04-caller.yml`.
   - This workflow has two inputs: `action` (GET/POST/PUT) and `env` (dev/prod).
   - It calls the template workflow by passing these inputs.

2. Open the template workflow at `./.github/workflows/04-task04-template.yml`.
   - Notice the `environment: ${{ inputs.env }}` line in the job definition.
   - This tells the job to use the environment variables from the environment passed in the input.
   - The script then reads `${{ vars.TARGET }}` and `${{ vars.MESSAGE }}` directly from that environment.

3. Navigate to GitHub and go to the `Actions` tab.
   - Select `LnD-GHA-Task04-Caller`.

4. Run the workflow with the following combinations and observe the outputs:
   - Action: GET, Environment: dev
   - Action: POST, Environment: dev
   - Action: PUT, Environment: prod

5. For each run, review the job output and note:
   - Which environment was used (check the `Run Task 04 Script` step output).
   - The URL and message that were resolved from the environment variables.
   - The response from the API call.

6. Compare the outputs from dev and prod runs:
   - Are the TARGET and MESSAGE values different between environments?
   - This demonstrates how the same workflow can behave differently based on environment-scoped configuration.

## Key Concepts

- **Reusable workflows** (`workflow_call`) allow you to define logic once and invoke it from multiple places.
- **Environment inputs** allow a workflow to select which environment's variables to use at runtime.
- **Environment variable resolution** (`${{ vars.VARIABLE_NAME }}`) only works when the job has set an `environment` context.
- This pattern is useful for promoting code through environments without duplicating workflow files.

Template files are more powerful than just this - for instance you could have a single caller workflow that calls multiple template files - this is very useful for multi-stage deployments; it can also be useful for Terraform, where you may have multiple subscriptions that need to be maintained across multiple tenants with `.state` files stored across many different Azure Storage Accounts. This is a very simple example, but provides an avenue to understanding the capabilities with GHA.
