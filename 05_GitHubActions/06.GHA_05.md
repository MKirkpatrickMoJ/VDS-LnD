## Scenario

You have seen how caller and template workflows work together and how environment inputs allow the same workflow to behave differently depending on the target environment. This task pulls those concepts together with a script you will complete yourself.

A starter script exists at `./05_GitHubActions/gha_scripts/05deployScript.ps1`. Starter workflow files also exist for both the template and caller. Your job is to complete all three files and run the finished workflow from GitHub Actions.

## Task

1. Open the starter script at `./05_GitHubActions/gha_scripts/05deployScript.ps1`.
   - Read the header to understand what the script should do.
   - The parameters are already defined — complete the body following the TODO comments.

2. Open the template workflow at `./.github/workflows/06-task05-template.yml`.
   - Define the `workflow_call` inputs to match the script's three parameters.
   - Set the job environment using the `env` input so that environment-scoped variables will resolve correctly (refer back to Task 04 for the pattern).
   - Add the step to call the script, passing the inputs as parameters.
   - The `$Application` parameter is defined as an environment variable within GitHub and should be called via ${{ vars.APP_NAME }}

3. Open the caller workflow at `./.github/workflows/07-task05-caller.yml`.
   - Define the `workflow_dispatch` inputs to match what the template expects.
     - `env` should be a `choice` input (dev / prod).
     - `version` should be a `string` input.
   - Add the job that calls the template workflow and passes the inputs through.

4. Commit and push all three files to your branch.

5. Navigate to GitHub and go to the `Actions` tab.
   - Select `LnD-GHA-Task05-Caller`.
   - Run the workflow with an application name and version, once targeting `dev` and once targeting `prod`.
   - Confirm the deployment tag in the output matches the expected format `<application>-<version>-<environment>`.

## Key Concepts

- The **template** (`workflow_call`) defines reusable logic. It does not know or care which caller invokes it.
- The **caller** (`workflow_dispatch`) defines how the workflow is triggered and what inputs it accepts at runtime.
- Inputs flow from caller → template, and the template uses them to configure its job and call the script.
- Setting `environment: ${{ inputs.env }}` on the job allows the template to resolve environment-scoped variables without the caller needing to know their values.
