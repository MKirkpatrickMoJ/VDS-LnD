# Final Notes

This is not a task, but a final note on GitHub Actions (GHA). This module has focused on running workflows manually via `workflow_dispatch` and triggering one workflow from another via `workflow_call`. Both are configured under `on` in a workflow YAML file.

One of the biggest strengths of GHA is that the same workflow logic can be triggered in different ways, depending on your delivery and operations needs.

## Schedules

A common requirement, especially for infrastructure, is to run housekeeping jobs on a regular schedule. Examples include cleaning up stale profiles, checking secret expiry dates, and identifying orphaned resources.

```yml
name: housekeeping
on:
  schedule:
    - cron: "15 9 2 * *"
```

In this example, the workflow runs at 09:15 on the second day of each month. Cron is a powerful way to run automation at fixed times or intervals. To learn the syntax, [Cronhub's crontab guide](https://crontab.cronhub.io/) is a helpful starting point.

## CI/CD Triggers

Continuous integration and continuous delivery are central to modern DevOps and PlatformOps practices. These triggers are commonly used to validate and release changes safely.

### Pull Requests

```yml
on:
  pull_request:
    types: [opened, synchronize, reopened]
    branches:
      - main
```
Here, this action would trigger is a Pull Request was created against the `main` branch - and would trigger again if any new commits are pushed after the PR was created, or if it was closed and re-opened. This is useful for things like running a Terraform plan and publishing the plan result as a comment for a reviewer to assess whether the code matches the intention, and it wouldn't be dangerous to add to production.

```yml
on:
  push:
    branches:
      - main
      - release/*
```

This runs whenever commits are pushed to `main` or branches that start with `release/`. Typical use cases are packaging, publishing artefacts, and deployment gates.

## Other Useful `on` Types

### Release Events

```yml
on:
  release:
    types: [published]
```

Useful when you want deployments or publication tasks to run only when a GitHub Release is published.

### Tag Pushes

```yml
on:
  push:
    tags:
      - "v*"
```

Useful for versioned releases, for example when tags like `v1.2.0` should trigger packaging or deployment.

### Chained Workflows

```yml
on:
  workflow_run:
    workflows: ["LnD-GHA-Task05-Caller"]
    types: [completed]
```

Useful for separating pipelines into stages, such as running security scans only after a build workflow completes.

### External System Trigger

```yml
on:
  repository_dispatch:
    types: [external-deploy-request]
```

Useful when another system (for example, a service desk or deployment tool) needs to trigger a workflow through the GitHub API.

## Summary

The `on` block defines when automation runs. Choosing the right trigger is as important as writing the job steps themselves. For most teams, a blend of `workflow_dispatch`, `workflow_call`, `pull_request`, `push`, and `schedule` provides a strong baseline, with `release`, `workflow_run`, and `repository_dispatch` added where needed.
